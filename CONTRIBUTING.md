# Contributing

感谢你为 `System Disk Slimmer` 贡献改进。

本项目是一个**以安全边界为优先、以规则配置为核心**的 Windows 清理工具。和一般“越删越多越好”的脚本不同，这个项目更强调：

- 先扫描，再决定是否清理
- 优先删除可重建缓存、日志、诊断文件、更新残留
- 规则要可解释、可追溯、可验证
- 单条规则宁可保守，也不要误伤系统或应用本体

在提交 PR 或提出规则建议前，请先阅读：

- `README.md`
- `RULE_AUTHORING_GUIDE.md`
- `GITHUB_RELEASE_CHECKLIST.md`

如仓库已启用 GitHub 模板，建议优先使用：

- `Bug report` issue 模板
- `Rule proposal` issue 模板
- `pull_request_template.md`

## 1. 你可以贡献什么

欢迎以下类型的贡献：

### A. 新的清理规则
适用于：
- 可重建缓存
- 日志文件
- 崩溃转储
- 诊断文件
- 更新残留
- 明确无运行时依赖的临时目录

### B. 现有规则修正
例如：
- 缩小规则范围
- 修正错误路径
- 补充 `Risk` 或 `RiskNote`
- 改进命名可读性

### C. 文档改进
例如：
- README 补充说明
- 新增截图或使用案例
- 完善发布流程
- 改进规则编写指南

### D. 国际化改进
例如：
- 修正文案
- 补充英文 / 日文翻译
- 统一术语表达

### E. 稳定性与验证改进
例如：
- 自检覆盖补充
- 测试建议
- GUI 行为一致性改进

## 2. 提交前的基本原则

### 规则 1：安全优先
禁止把以下对象直接作为“可删目标”提交，除非已经有非常强的系统性证据，并且维护者明确接受：

- 系统核心目录
- 应用主程序目录
- 驱动、运行时、共享组件
- 安装器缓存中仍可能被当前版本依赖的内容
- 无法确认用途的数据目录

### 规则 2：单条规则尽量窄化
优先：
- 精确到具体应用
- 精确到具体缓存 / 日志目录
- 精确到具体文件模式

避免：
- 一条规则横扫整棵大目录
- 用过宽泛的通配去覆盖未知内容
- 把“可能是缓存”的目录整体纳入删除

### 规则 3：先给证据，再给规则
新增规则时，请说明：
- 目标应用名称
- 路径来源
- 文件大致类型
- 为什么判断其可安全重建
- 是否实际观测过清理后应用可正常恢复

### 规则 4：不要顺手扩大战果
如果你只验证了一个缓存目录，就只提交这个缓存目录；不要顺手把同级其他未知目录一起加进去。

## 3. 建议贡献流程

### 方式一：提交 Issue / 讨论
适合：
- 你发现了新的候选目录
- 你不确定是否安全
- 你想先讨论风险边界

推荐：
- Bug 类问题使用 `Bug report` 模板
- 新规则建议优先使用 `Rule proposal` 模板

建议提供：
- 应用名称与版本（如能提供）
- 目录路径
- 目录大小或文件数量
- 样例文件名
- 你的安全判断依据
- 清理后是否会自动重建

### 方式二：直接提交 Pull Request
适合：
- 规则范围明确
- 已完成本地验证
- 文档和翻译已同步更新

推荐：
- 提交 PR 时使用仓库内置 `pull_request_template.md` 检查变更完整性

PR 建议包含：
1. 变更摘要
2. 为什么需要这条规则
3. 风险判断
4. 本地验证结果
5. 是否运行过 `-SelfTest`

## 4. 本地修改建议

### 需要关注的主要文件
- `system_disk_slim_gui.ps1`：GUI 与扫描 / 清理主逻辑
- `cleanup_targets.json`：清理规则定义
- `translations.json`：多语言文案
- `RULE_AUTHORING_GUIDE.md`：规则编写规范

### 新增规则时通常要做的事
1. 在 `cleanup_targets.json` 中新增或调整规则
2. 如新增 `RiskNote`，同步更新 `translations.json`
3. 确认命名、风险级别、路径范围符合规范
4. 运行自检
5. 必要时更新 `README.md` 或 `CHANGELOG.md`

---

## 5. 提交前检查

建议至少完成以下检查：

- [ ] 规则范围足够窄，没有误删明显风险
- [ ] `Name` 可读、可追溯
- [ ] `Type` / `Path` / `Paths` / `Include` 等字段正确
- [ ] `Risk` 与实际影响相符
- [ ] `RiskNote` 已在 `translations.json` 中覆盖
- [ ] 未引入机器私有路径或个人目录假设
- [ ] 运行过以下命令并通过：

```powershell system_disk_slim_tool/CONTRIBUTING.md
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\system_disk_slim_gui.ps1 -SelfTest
```

## 6. 规则命名建议

推荐命名风格：

- `Observed <App> <Target> Round <N>`
- 或简洁描述型命名，但要能表达“对象 + 内容类型”

例如：
- `Observed Edraw MindMaster QtWebEngine Cache Round 20`
- `Observed LarkShell Aha Profile Cache Round 19`

命名目标：
- 看名字就知道清理对象
- 能回溯到观察来源或整理轮次
- 避免模糊名称如 `Misc Cache 1`

更详细规则见：`RULE_AUTHORING_GUIDE.md`

## 7. 关于翻译与文档同步

如果你的变更影响以下内容，请同步更新文档：

- 新增 UI 文案
- 新增 `RiskNote`
- 调整运行方式
- 调整风险边界
- 新增对外可见功能

至少应考虑更新：
- `translations.json`
- `README.md`
- `CHANGELOG.md`

---

## 8. 不建议提交的内容

请不要提交以下内容：

- `exports/` 下导出产物
- 扫描结果 CSV / JSON
- 导出的日志文件
- 机器私有临时脚本
- 与项目无关的本地测试文件
- 绝对路径、用户名、设备特有信息

## 10. 维护偏好总结

维护上优先接受的贡献通常具备以下特征：

- 有证据
- 范围窄
- 风险说明清楚
- 能通过自检
- 文档同步完整
- 不引入环境绑定和路径假设

如果你不确定一条规则是否安全，**先提 Issue / 先讨论，再提交 PR**，通常会比直接扩大量级更合适。
